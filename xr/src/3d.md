# `xr/src` Refactor Plan

## Goal

Remove `XrScene` and rebuild `XrRoot` as the single scene host. Add `XrView` for hosting 2D UI inside 3D space.

## Current State (before refactor)

```
App
 └─ XrRoot (shell: desktop preview, XR mode, panel placement, orbit camera)
      ├─ XrEnv (environment draws: depth mesh, passthrough, PBR, env cube)
      ├─ XrScene (scene host: physics, XrNode tree walk, runtime bodies)
      │    └─ XrNode children (Cube, Gltf, Tree, RefractiveCube, …)
      └─ controls widget (2D UI panel rendered into 3D via panel_matrix)
```

`XrScene` sits between `XrRoot` and the content. It owns:
- Rapier physics scene (`RapierScene`)
- runtime body state (`runtime_bodies`)
- cube collection from the widget tree
- physics step loop (both desktop preview and XR)
- `draw_3d` delegation to child nodes

`XrRoot` owns:
- desktop orbit camera + presentation state
- XR panel matrix computation + XR interaction routing
- scene switching (template_widgets, scene_order, active_scene)
- pass/window setup
- `XrEnv` (which does env draws but also calls into `XrScene` for physics state)

## Target State (after refactor)

```
App
 └─ XrRoot (scene host + shell)
      ├─ XrEnv (owned, not a child widget — provides physics + env draws)
      ├─ named children drawn in order (Cube, Gltf, Tree, XrView, …)
      └─ z-sorted primary drawlist
```

### XrScene: removed

All of `XrScene`'s responsibilities move into `XrRoot`:
- Physics ownership (`RapierScene`, runtime bodies) → `XrRoot` via `XrEnv`
- Cube collection + `rebuild_runtime_scene` → `XrRoot` or `XrEnv`
- `draw_3d` delegation → `XrRoot` draws children directly
- `on_render` / `script_call(render)` → `XrRoot` calls render on itself

### XrRoot: rebuilt

`XrRoot` becomes the single scene host. It:

1. **Owns an `XrEnv`** (not as a child widget, but as an owned struct field). `XrEnv` provides:
   - Physics scene (Rapier): create, step, sync bodies
   - Environment draws: depth mesh, passthrough env atlas, PBR setup, env cube
   - Hand rendering (optional debug geometry)

2. **Has named children** via `:=` syntax in script_mod. These are the scene content widgets (Cube, Gltf, Tree, XrView, etc.). They are drawn in declaration order.

3. **Z-sorts its primary drawlist**. After all children have drawn into the 3D drawlist, `XrRoot` sorts draw calls by distance from camera before submitting. This gives correct transparency ordering without per-widget sorting.

4. **Keeps existing shell duties**:
   - Desktop orbit camera + preview pass
   - XR mode pass setup
   - XR panel matrix for UI overlay
   - Scene switching (if we keep multi-scene support, or simplify to single scene)

### XrView: new widget

`XrView` hosts a 2D makepad UI inside 3D space. It:

- Has an `XrNode` (position, rotation, scale in the scene)
- Creates a `Cx2d` from the `Cx3d` and uses a `DrawList2d` with the right view transform matrix (computed from the node's world pose + pixel scale). No offscreen texture — the 2D draw calls go directly into the 3D pass with a transformed draw list.
- Routes XR finger-tip events into 2D hit-testing on the plane
- Replaces the current `controls` panel mechanism in `XrRoot` (which was hardcoded panel_matrix logic)

In the example app (`examples/xr/src/main.rs`), `XrView` is used like:
```
block_ctrl := XrView{
    pos: vec3(-0.92, 0.42, -0.62)
    look_at: vec3(-0.10, 0.32, -0.62)
    logical_size: vec2(320, 400)
    pixel_scale: 0.0004
    RoundedView{
        // 2D UI content — drawn directly into the 3D pass
        // via a DrawList2d with the right view transform
    }
}
```

## Refactor Steps

### Step 1: Move physics into XrEnv

`XrEnv` already owns the draw objects (draw_cube, draw_pbr, draw_depth_mesh, draw_passthrough_env_atlas). Add to it:
- `RapierScene` (currently on `XrScene`)
- `runtime_bodies` (currently on `XrScene`)
- `collect_cubes_from_widget` + `rebuild_runtime_scene` (currently on `XrScene`)
- Physics step + sync (currently split between `XrScene::handle_event` and `XrEnv::prepare_draw_scope`)

`XrEnv` becomes the complete scene-environment owner: physics + rendering environment.

### Step 2: Remove XrScene

- Delete `scene/xr_scene.rs`
- Remove from `lib.rs` module list and `script_mod` registration
- Remove `pub use xr_scene::XrScene`

### Step 3: Rebuild XrRoot from scratch

The current `XrRoot` is ~1300 lines of tangled concerns. Nuke it and write a clean one.

**What XrRoot IS**: a Window that sets up a 3D pass with a camera, draws its named children into that pass, and z-sorts the result. That's it.

**What XrRoot is NOT**: a scene switcher, a panel hit-tester, a controls host, an orbit camera manager, or an XR interaction router. Those things either move to `XrView` or become app-level concerns.

Clean `XrRoot` struct fields:
```rust
pub struct XrRoot {
    // Widget basics
    #[walk] walk: Walk,
    #[layout] layout: Layout,
    #[rust] area: Area,

    // Scene environment (physics + env draws)
    #[live] env: XrEnv,

    // Pass / window (XrRoot IS a window)
    #[live] window: ScriptWindowHandle,
    #[live] pass: ScriptDrawPass,
    #[new] depth_texture: Texture,
    #[new] main_draw_list: DrawList2d,
    #[new] xr_draw_list: DrawList,

    // Desktop camera (simple orbit)
    #[live(28.0)] camera_fov_y: f32,
    #[live(3.4)] camera_distance: f32,
    #[live(0.05)] camera_near: f32,
    #[live(200.0)] camera_far: f32,
    #[rust(0.0)] orbit_yaw: f32,
    #[rust(0.45)] orbit_pitch: f32,

    // Children (named widgets from := declarations)
    #[rust] children: Vec<(LiveId, WidgetRef)>,
}
```

Clean `XrRoot` behavior:
1. `on_after_apply`: collect `:=` children into `self.children` vec (preserving order)
2. `handle_event`:
   - Forward events to `env` and all children
   - Handle desktop orbit (mouse drag on background = orbit, scroll = zoom)
   - On `Event::XrUpdate`: forward to env for physics step
3. `draw_walk` (desktop path):
   - Set up pass with orbit camera projection
   - Call `env.prepare_draw(cx)` for env rendering
   - Iterate children, call `draw_3d_all` on each
   - Z-sort the drawlist
4. `draw_3d` (XR path):
   - Same but camera comes from XR headset pose

**Things that are GONE from XrRoot**:
- `controls` / `controls_widget()` — use XrView instead
- `scene` / `active_scene` / `scene_order` / `switch_scene` / `select_scene` — app toggles `visible` on children
- `template_widget()` / `active_scene_widget()` — just iterate `self.children`
- `xr_panel_pixels` / `xr_dpi_factor` / `xr_pixel_scale` / `xr_depth_scale` — moves to XrView
- `xr_position_offset` / `xr_forward_offset` / `desktop_panel_position` / `desktop_panel_look_at` — moves to XrView
- `compute_panel_matrix_from_pose` / `panel_pose_from_head_pose` — moves to XrView
- `desktop_panel_hit` / `desktop_panel_contains` / `desktop_panel_active` — moves to XrView
- `make_desktop_xr_local_event` / `dispatch_desktop_xr_local_event` — moves to XrView
- `xr_toggle_with_menu` / `xr_visible` — per-widget concern (XrView or app-level)
- `scene_anchor_to_head` / `scene_anchor_*` — moves to XrEnv or app-level
- `xr_view_matrix` / `xr_hit_matrix` / `xr_view_matrix_initialized` — moves to XrView
- All the `xr_panel_log_count` / `xr_draw_logged` / `scene_anchor_log_count` debug noise
- `DesktopPresentationState` struct — simplified inline
- `DrawState` enum — unnecessary for a simple single-pass draw

The desktop orbit camera is ~40 lines. The pass setup is ~30 lines. The child iteration is ~15 lines. The whole file should be ~200-300 lines, not 1300.

### Step 4: Create XrView

New file `obj/xr_view.rs`:
- Struct with `XrNode` (for 3D placement), `DrawList2d`, pixel scale, logical size
- `draw_3d`: creates `Cx2d` from `Cx3d`, sets the draw list's view transform matrix (node world pose × pixel-to-world scale), draws 2D children into it with a root turtle at the logical size. No offscreen texture — draw calls go straight into the 3D pass.
- `handle_event`: converts XrLocal finger-tip events into 2D coordinates on the plane, forwards to 2D children. Also handles desktop mouse ray-casting for the preview.

### Step 5: Update examples/xr/src/main.rs

- Remove `XrScene` wrapping
- Place content directly as `XrRoot` children
- Replace the old `controls` panel with `XrView`
- Scene switching becomes toggling `visible` on child nodes

## Open Questions

- **Z-sort granularity**: Sort at the draw-call level within the DrawList? Or just sort the order children are drawn? Draw-call level is more correct for intersecting geometry but may need DrawList API support. Child-level is simpler and probably good enough.

- **XrView desktop interaction**: XrView needs to handle both XR finger-tips AND desktop mouse ray-casting (for the desktop preview). The ray-cast math is straightforward (project mouse through orbit camera, intersect with the quad plane). Should XrView own this, or should XrRoot provide a helper that converts mouse events into ray events that XrView consumes?
